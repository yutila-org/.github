# Security Policy

Yutila prioritizes system integrity and secure architectural design. Our complete framework for vulnerability management, access control, and incident response is detailed in our [Security Policies](https://docs.google.com/document/d/1zqEZ9wyOiUj6hyH294iyZh9IXi8PnqnQC8UB7tnc0I0/edit?usp=sharing).

## Supported Versions

Security updates are actively prioritized for all active repositories within the organization.

## Reporting a Vulnerability

If you discover a potential vulnerability within any Yutila repository, do not disclose it publicly. 

1. **Primary Method:** Use the **"Report a vulnerability"** feature located under the **Security** tab of the affected GitHub repository.
2. **Backup Method:** If that feature is unavailable or the vulnerability impacts organizational infrastructure across multiple repositories, email full details to **yutila@atomicmail.io**.
3. **Details Required:** Provide a precise description of the exploit, proof-of-concept code, and the specific architecture affected.
4. **Response Protocol:** The Incident Commander will triage the report according to our internal matrix:
   * **Critical/High:** Immediate Incident Response Team (IRT) mobilization. Expect an initial response within 24 hours.
   * **Medium/Low:** Scheduled for the next technical debt refactoring cycle.

All reports trigger an evaluation against our Security Policies. A post-mortem and remediation plan will be formulated for all validated vulnerabilities.
